# 自作 macOS IME「RomKana」開発記（続）— 文節変換と個人最適化

[前回の開発記](https://zenn.dev/toshinao/articles/1cffb713b1c670)では、ローマ字をローカルでかな漢字に変換する macOS の IME を、辞書＋ローカルLLM（第1部）から **AzooKeyKanaKanjiConverter ＋ Zenzai**（第2部）へ作り直すところまで書きました。毎日使える道具にはなったのですが、使い込むと「もっとこうしたい」が2つ出てきました。

- **(a) 空白で区切らずに長文を打つと、一部分だけを直せない。**
- **(b) 社内用語や固有名詞を、その都度ユーザー辞書に手で足している。**

この2つを潰した記録です。現行構成のリファレンスは [`architecture.md`](https://github.com/t10471/RomKana/blob/master/docs/architecture.md) を参照してください。

---

## 1. 文節変換 — 空白なしでも一部だけ選び直す

前回（第2部・11章）で「空白で区切る＝文節境界を教えてもらって、1語だけ差し替える」はできていました。でも空白なしで長文を一発変換したとき、途中の1文節だけ間違っていると、文全体の別候補から選び直すしかありません。標準的な日本語IMEのように「**←/→ で文節を選び、その文節だけ変換し直す**」をやりたくなりました。

AzooKey の中を調べてみると、**文節分割の出し方が一筋縄ではない**と分かりました。

- ❌ 最初は `firstClauseResults`（先頭文節の候補）を取り、`prefixComplete` で前から1文節ずつ切り出す方式にしました。きれいに分かれる…はずが、「きょうははいしゃにいく」が**まるごと1文節「今日は歯医者に行く」**になってしまうことがある。これだと → を押しても動きません（そもそも分かれていない）。
- ✅ そこで、文全体のベスト候補の**構成要素（単語列）**で区切る方式に変えました。各単語がカバーする読みの文字数で、読みをスライスして文節にする。これなら複数語の文は必ず複数文節に分かれます。

🔑 もう一つ、AzooKey には**「文節の境界を伸び縮みさせる API」が無い**ことも分かりました。変換は常に文全体に対して走ります。なので、文全体の読みを自前で「文節ごとの区切り位置」として持ち、境界を動かしたら**影響する文節だけ読みを再変換する**形にしました。第2部で作った空白チャンク変換の仕組みが、ほぼそのまま流用できたのは助かりました。

### ターミナルでフォーカスが見えない

動くようになって喜んだのも束の間、**WezTerm で使うとどの文節を選んでいるか分からない**。調べると、macOS では WezTerm の preedit 描画設定が効かず、こちらが付けた文節区切り（`markedClauseSegment`）・背景色・下線の太さを**ターミナルは反映しない**のでした（GUIアプリなら出ます）。割り切って、フォーカス文節を `《…》` で囲んで表示することにしました。端末でもはっきり分かりますし、確定する文字に記号は入りません。

### 「→を押すと消える」の正体

使ってみると「最後まで打って → を押すと全部消える」という現象が出ました。ログを仕込んで再現したら、**押していたのは Shift+→ でした**。Shift+Space で変換した直後、Shift を押したまま → を押すと、当時 Shift+←/→ に割り当てていた「境界伸縮」が暴発していました。先頭文節が隣をどんどん吸収して、最後は1文節に統合されます。消えていたのではなく、まとまっていたのです。なので **フォーカス移動は ←/→（Shift の有無に関係なく）**、**境界伸縮は Option+←/→** に変えました。変換キー（Shift+Space）の修飾が指に残っても誤爆しません。地味ですが、「変換キーと編集キーの修飾がぶつからないように割り当てる」のは大事だと学びました。

### 「聞いて返ってきた」が「帰って」にしかならない

毎日使っていると、自分がいちばんのテスターになります。空白なしで `きいてかえってきたもの` と打つと、`かえって` を `返って` に選び直せない、と気づきました。掘ってみると原因が3つ重なっていました。

- **文節の境界がずれる。** ベスト候補の構成要素で区切る方式だと、`聞いて帰ってき` のように **te形の連なりが1要素に癒着**して、`かえって` が独立した文節になりません。`firstClauseResults` で切り直そうとしても、今度はこの節の冒頭で書いた「まるごと1文節」の崩壊が起きます。
- **Zenzai が同音異字を落とす。** `かえってきた` 単体を変換しても、Zenzai は「入力全体をカバーする候補」に `返ってきた` を残しません。ところが **Zenzai を切った辞書ラティス単体なら `返ってきた` が候補に出ます**。
- **しかも出たり出なかったり。** 最初の直しでは「癒着しているか」を Zenzai のベスト候補で判定していたのですが、Zenzai はサンプリングで毎回わずかに違う答えを返すので、**同じ入力でも分割が変わる**気持ち悪い挙動になりました。

直し方はこうです。**癒着の判定は、毎回同じ答えになる辞書ラティス（Zenzai オフ）で行う**。再分割は、**漢字のかたまりがちょうど1つになる「いちばん長い前方」で切る**（`聞いて帰って` は漢字が2つ＝2文節なので却下し、`聞いて` で切る）。そして文節ごとの候補は **Zenzai あり・なしの両方をマージ**して、Zenzai が落とした `返ってきた` も選べるようにしました。正常な文はこれまでどおり Zenzai の文脈変換に任せ、癒着を検出したときだけ再分割します。

## 2. 個人最適化 — 自分の語彙に寄せる

社内用語や固有名詞（社内の略語や製品名など）を、その都度ユーザー辞書に手登録していました。これを「使っているうちに勝手に賢くなる」形にできないか。

調べてみると、**AzooKey/Zenzai には個人最適化の仕組みが最初から入っていました**。`convertOptions` で `personalizationMode: nil` にしていた、まさにそこです。仕組みは、自分の文章から作った**個人 N-gram モデル**を、汎用の **base N-gram** で割った比（＝自分はよく使うが世間では珍しい言い回し）で、Zenzai の候補を持ち上げる、というもの。本家の [azooKey-Desktop](https://github.com/azooKey/azooKey-Desktop) も同じ仕組み（Tuner）で個人最適化しています。

実装で1つ制約にぶつかりました。**学習関数（`trainNGram`）を持つモジュールが外部公開されていない**のです。推論側（個人モデルを使う）は `personalizationMode` にファイルのパスを渡すだけで動く（変換器が中で読んでくれる）のに、学習だけは本体から直接呼べない。なので学習は、AzooKey の CLI ツールを `swift run` で別プロセスとして回す**スクリプト**に分けました。流れはこうです。

1. 確定した文を履歴ファイルに貯める（個人最適化を ON にしたときだけ・平文ローカル）。
2. ある程度貯まったら、スクリプトで個人 N-gram モデルを学習する。
3. base モデル（[Hugging Face にある n=5 の汎用 N-gram](https://huggingface.co/Miwa-Keita/base_n5_lm)）と合わせて有効化する。

N-gram なので軽い（数十MB）し、本体モデルを再学習するのと違って**汎用の変換性能を壊すリスクが無い**（混ぜる強さを `alpha` で調整できる）のが気に入っています。前回の第1部で「賢いのはモデルより使う人の癖」と書きましたが、その癖を今度は N-gram という形で素直にモデルへ取り込めるようになった、という感覚です。

正直なところ、効果がはっきり出るには履歴がそれなりに要りますし、base モデルが HF にライセンス明記が無い（なので個人利用に留め、リポジトリには含めず取得手順だけ置いた）といった注意点もあります。まだ育て始めたばかりです。

## 今回の教訓

- **ライブラリは公開APIの境界で設計が決まる。** 個人最適化は推論こそパスを渡すだけで有効化できたのに、学習関数は非公開で、CLI を別プロセスで回す形に分けることになりました。「使えると思った機能が、半分しか公開されていない」のはよくある話で、境界を最初に確かめるべきでした。
- **ターミナルとGUIで marked text の扱いが違う。** 文節のフォーカスは属性（下線・背景・clause segment）で見せるとGUIでは出るが端末では消える。記号（`《》`）で囲むのが結局いちばん確実でした。
- **変換キーと編集キーの修飾はぶつけない。** Shift+Space 変換の Shift が指に残ったまま矢印を押すと境界伸縮が暴発した。フォーカス移動は ←/→、伸縮は Option+←/→ に分けて解決しました。
- **設計の判断材料にする出力は、決定的かどうかを確かめる。** 文節が癒着しているかの判定を Zenzai（サンプリングあり）の結果でやったら、同じ入力で分割が揺れました。判定を辞書ラティス（毎回同じ答え）に寄せたら安定しました。賢い出力ほど、揺れることを忘れがちです。
- **「賢いのは使う人の癖」は今も同じ。** 第1部ではユーザーの選択履歴で同音語を分岐させ、今回はその癖を個人 N-gram としてモデルに渡す。アプローチは変わっても、効くのは結局「その人のデータ」でした。
