# 自作 macOS IME「RomKana」開発記 — ローカルLLMでローマ字をかな漢字に変換する

ローマ字を打つと、文脈を見て自然なかな漢字交じり文に変換する macOS の入力メソッド（IME）を、**完全ローカル**で動くように自作した記録です。変換の賢さをローカルLLMで底上げできるか、を実測しながら詰めていきました。

例: `kouyuhuuninyuuryokusuruto` → `こうゆう風に入力すると`

この記事は「やったこと・分かったこと・つまずいた所」を、後から人に説明できる形で全部まとめたものです。

---

## 1. 何を作ったか

- **RomKana.app**: Swift + InputMethodKit で書いた macOS 用の本物の IME（システム設定の入力ソースに出る）
- **変換サービス**: Python の HTTP サーバ（`127.0.0.1:8765`）。辞書変換 + 任意で LLM リランク
- 配布先: `~/Library/Input Methods/RomKana.app`、サービスは LaunchAgent で常駐

### アーキテクチャ

```
[ローマ字キー入力]
   │ IMKInputController が捕捉
   ▼
RomKana.app (Swift / IMK)
   ├─ ローカル romaji→かな表で即時インラインプレビュー（同期・無遅延）
   └─ Space で変換要求 ── localhost HTTP ──▶ 変換サービス (Python)
                                              ├─ mozcpy（辞書）で n-best 候補生成
                                              └─ 任意: ローカルLLMで候補をリランク
   ▼ 候補をポップアップ表示、Enter で確定
```

**二層構成が肝**: 打っている最中のかな表示はローカル表で即時。重い変換は Space 押下時だけ。

---

## 2. 第1の山 —「IMEがシステムに登録されない」

最初の最大の難所は、変換でも何でもなく **「作った IME が入力ソースの一覧に出てこない」** ことでした。署名を変え、何度も再ログインし、ログを追っても出ない。数日相当ハマりました。

### 真因: バンドルIDに `.inputmethod.` が必要だった

macOS は入力メソッドを認識するのに、**`CFBundleIdentifier` に `.inputmethod.` という部分文字列が含まれること**を要求します。

- ❌ `com.toshinao.romkana` → 出ない
- ✅ `com.toshinao.inputmethod.RomKana` → 出る

Apple 純正も `com.apple.inputmethod.Kotoeri`、Google も `com.google.inputmethod.Japanese`。この規約を満たした瞬間、`TISRegisterInputSource` が実際に登録するようになりました。

### 切り分けで「ハズレ」だった容疑たち（全部無関係）

- **署名/公証は無関係**: ad-hoc 署名でも IME は動く（参考にした VoiceDict が ad-hoc + `spctl rejected` でも動作）。Apple Development 証明書も公証($99)も不要だった
- **キャッシュ汚染でもない**: バンドルIDを新品にしても直らなかった（真因は別）
- **Info.plist の `ComponentInputModeDict` は必須ではない**: 動作サンプル(azooKey)は持っていなかった

### さらにハマった細部

- 入力モードのキーは **Apple 標準名 `com.apple.inputmethod.Japanese`** にする（Google日本語入力と同じ）。独自名にすると「カナ入力」扱いになったり表示名が生のIDになる
- 表示名は `Resources/*.lproj/InfoPlist.strings` でローカライズ（`"com.apple.inputmethod.Japanese" = "ひらがな (RomKana)"`）
- メニューアイコンは 16/32px の小さい tiff（128px だと巨大表示になる）
- 候補ウィンドウが画面に出ない問題は、Typut から拝借した `setWindowLevel:` の `perform` ハック（構造体を整数引数に渡す不正呼び出し）が原因で、**削除したら直った**

**教訓**: IME 開発の 9 割は「変換」ではなく「OSにどう認識させるか」の儀式。

## 3. 第2の山 — 変換エンジン

### ベースは mozcpy（Mozc 派生の辞書変換）

最初にローカルLLM単体（LFM2.5-1.2B-JP）で変換させたら、文脈無視・チャット応答が混ざるなどで実用外。そこで **mozcpy（辞書）を主**にしました。速い（~20ms）・安定・ローカル。

### 口語正規化ルール

崩れた口語ローマ字を、辞書を引く前に正規化:

- `こうゆう→こういう`、`そうゆう→そういう`、`とゆう→という` など
- 正規化した読みと元の読みの両方の候補をマージ（`交友風` も残しつつ自然な `こういうふうに` を上位に）

### 外来語・記号・末尾「ん」

実際に打ってみて出たバグを潰しました:

- 外来語の音 `じぇ(je/zye)`・`ちぇ(che)`・`てぃ(thi)` 等が表に無く、`プロジェクト` が `ぷろzyぇくと` になっていた → 表に追加
- 末尾の単独 `n` を「ん」に確定していなかった（`shinbun`→しんぶ**n**→変換崩壊）→ 変換時だけ末尾 n→ん
- `[` `]` → 「 」 などの記号変換、記号で合成が切れない処理

## 4. UX の作り込み（実打鍵で出た問題を1個ずつ）

ドッグフーディングしながら直した主なもの:

- **候補ポップアップ（IMKCandidates）**: 一覧が見えず盲目的に循環していたのを、ウィンドウ表示＋巡回選択に
- **Space の挙動**: 「変換 → もう一度 Space で次候補へ循環、Enter で確定」（Google日本語入力と同じ）。何度も「2回目のSpaceで候補を行き過ぎて確定」事故が起きたので調整
- **Backspace**: ローマ字1文字ずつ消すと途中で「k」が見える → **かな単位で削除**
- **英語混在**: 英数⇄かなトグル＋「英字＋Space→候補に LLM/llm が出る」両対応
- **プレビューが漢字に変わらないバグ**: 非同期処理内で `self.client()` が nil になり `setMarkedText` がスキップされていた。**変換を同期化**して handle() 内で確定セット

## 5. 第3の山 — 「ローカルLLMで変換を賢くできるか」を実測

ここからが本題。同音異義語を文脈で正しく変換できるか、を多数のモデルで測りました。

### テストセット

文脈依存の同音異義を中心に、易しめ10問と本格的な30問を用意:

- 易: `へんかんがびみょう→変換が微妙`
- 難（文脈依存）: 文脈「そらが曇って」+`あめ`→**雨** / 文脈「子供が喜ぶ」+`あめ`→**飴**、記者/汽車、対象/対照、精算/清算、科学/化学、意思/医師 …
- 略語: `おねしゃす→お願いします`、`とりま→とりあえず`

### 重要な指標の発見:「候補に含まれる率」

IME は**ポップアップから選べる**ので、本当の指標は「top-1 が正しいか」ではなく「**正解が候補一覧に含まれるか**」。

- 辞書 top-1: **15/30**
- だが**正解が候補内にある率: 27/30**（多くは1〜4位）。残り3つは略語（読み自体が辞書に無い）

→ つまり**辞書＋ポップアップで実用上かなり良い**。top-1 の数字は実態より厳しく見えていた。

### 各モデルの実測（難所30問・SELECT＝候補から選ばせる方式）

| モデル | サイズ | SELECT正解 | 速度 |
|---|---|---|---|
| mozcpy（辞書） | 小 | 15/30（候補内27） | ~20ms |
| Qwen2.5-7B | 4.3GB | 15-16/30 | ~1000ms |
| Gemma4-E2B | 4.9GB | 15/30 | ~640ms |
| Qwen3-1.7B | 1GB | 15/30 | ~480ms |
| Sarashina2.2-3B | 2GB | 13-14/30 | ~600ms |
| ELYZA-8B | 5GB | 11-13/30 | ~1000ms |
| Qwen3-4B | 3GB | 6/30 | ~900ms |
| Llama-3.2-3B | 2GB | 3/30 | ~450ms |
| LFM2.5-1.2B | 0.8GB | 1/30 | ~200ms |

**結論**: SELECT/GEN（指示に従わせる方式）では、**大小どのオフザシェルフ LLM も辞書(15)を超えない**。賢いモデルですら「辞書のtop候補をなぞるだけ」。
参考: クラウドの **Opus 4.8 は ~29/30**（文脈推論が桁違い。Sumibi 作者のベンチでもクラウド Gemini が1位）。

## 6. ファインチューニング PoC（Sarashina-3B + LoRA）

「汎用が無理なら専用化」。MLX の LoRA で Sarashina-3B を IME タスクに微調整。

- データ: 約2,500件（同音異義ターゲット＋現代コーパス＋略語、**テスト30問は不使用**）。読みは SudachiPy で自動付与
- 訓練: 4bit QLoRA、約30分、loss は 6.4→0.5 と健全に収束

### 結果

| 方式 | 素 → FT |
|---|---|
| **GEN（生成）** | 5/30 → **9/30**（+80%、効いた） |
| **SELECT（選択）** | 13/30 → **9/30**（悪化） |

- **FT 自体は効く**（GEN が約1.8倍）。ただし辞書(15)には未到達、生成型は造語(製産/整算)のハルシネーションが残る
- **SELECT-FT は逆効果**。原因はコーパス由来ラベルのノイズ。SELECT-FT は**ラベル品質に極めて敏感**で、雑なデータは害になる
- Gemini の「300件で25-30点」は楽観的だった。辞書超えには「大量・クリーン・人手検証済み」データが必須

## 7. ブレイクスルー — 「LLMスコアリング方式」

ここで方針転換。LLM に**指示で選ばせる/生成させる**のをやめ、LLM 本来の「**次に来る自然な日本語の確率計算**」だけを使う:

1. 候補生成は**辞書(mozcpy)に丸投げ**（候補は実在＝**ハルシネーション物理的にゼロ**）
2. 各候補について `P(候補トークン | 文脈)` を**1回の forward** で計算
3. **合計対数尤度（sum）が最大**の候補を採用

### 結果（難所30問・sum scoring）

| モデル | SCORE-sum | 速度 |
|---|---|---|
| 辞書（基準） | 15/30 | ~20ms |
| **Qwen2.5-7B** | **19/30** | ~860ms |
| **Qwen3-1.7B** | **17/30** | ~410ms |
| **Qwen2.5-1.5B-base** | **16/30** | ~235ms |
| Gemma2-2B-jpn-it | 16/30 | ~396ms |
| Qwen2.5-0.5B | 14/30 | ~105ms |
| Sarashina-3B | 5/30 | ~435ms |

**初めて辞書(15)を突破。** 生成しないので速く、造語もゼロ。

### 効いた理由・効かない理由（ここが面白い）

- 🔑 **sum が正解、mean はダメ**（mean だと11止まり）。自然で高頻度な漢字語ほど**少ないトークンにまとまる**ため、合計尤度がそれを味方につける。平均はその信号を打ち消す。**初期に「LLMリランクは効かない」と封印したのは、mean でやっていたから**＝早すぎた棄却だった
- **scoring に向くのは「ベースLMの確率が素直で日本語Web知識が厚いモデル」** → Qwen が好適。**Sarashina は SELECT では13点なのに scoring では5点**と相性が逆転（対話チューニングやトークナイザ境界の影響と推測）
- **chat テンプレ縛りで除外していたモデルが復活**: Gemma 系は「system role 非対応」で SELECT 評価が落ちたが、scoring は生テキストを食わせるだけなので無関係
- 0.5B は ~105ms と爆速だが **14/30＝辞書未満**で不採用

## 8. この時点でのアーキテクチャ

```
ローマ字 → [ローカルかな表] → かな読み
                                  │ Space
                                  ▼
                  mozcpy（辞書）＋口語正規化 → n-best 候補（実在・安全）
                                  │ use_llm=true
                                  ▼
        Qwen（1.5B-base〜7B）で sum-loglik リランク（文脈条件付き・生成ゼロ）
                                  ▼
                      ポップアップ先頭が文脈的に賢くなる
```

- 採用モデルは1行差し替えで速度/品質を調整: **Qwen2.5-7B(19/30) / Qwen3-1.7B(17) / Qwen2.5-1.5B-base(16・~235ms)**
- 確定済みテキストを文脈として渡し、雨/飴・精算/清算 を文脈で出し分け（実機で確認済み）

### 残課題

- **同期処理ゆえ Space ごとに 0.2〜1秒のフリーズ**。→ 次は**非同期化**（押した瞬間に辞書結果を即表示、裏でLLMが並べ替え＝待ち時間ゼロ）
- デバッグログ等の一時コード除去・本番構成化

ここから先は、変換の芯ができた後に「実際に毎日使えるUX」へ仕上げた記録。Space ごとのフリーズを消し、スコアリングを速くし、自分の語彙に寄せ、Sumibi 風の打ち方に対応した。

## 9. 「フリーズ」をどう消すか — ジャンプとの戦い

同期変換だと Space のたびに 0.2〜1秒固まる。これを消したい。

- **最初の案（非同期）**: Space で辞書結果を即表示し、裏で LLM が先頭を並べ替える。→ **1番目が後から入れ替わって気持ち悪い**（ジャンプ）。
- **案2（先頭ピン留め）**: #1 は辞書のまま固定、LLM は2位以降だけ並べ替え。ジャンプは消えるが、**#1 が LLM 品質に戻らない**（辞書の15/30に逆戻り）。
- **結論（上限つき同期待ち）**: Space で辞書と LLM を**並行起動**し、**最大400msだけ LLM を待つ**。間に合えば LLM 順で表示、超えたら辞書順で確定して**以後は並べ替えない**。

これで「ジャンプ無し・#1 に LLM 品質・体感 ~0.1〜0.3秒」に落ち着いた。

**教訓**: 「待ち時間ゼロ」と「1番目が動かない」は**同時に成立しない**（最初に出すものが最終結果でない以上、必ずどちらかを諦める）。逃げずにトレードを直視し、「待ちを十分小さくして同期で待つ」方向で解いた。そのために次の高速化が要った。

## 10. スコアリングを約2.5倍速くする — バッチ化とKVキャッシュ

「同期で待つ」を成立させるには、リランクを速くする必要がある。元の実装は無駄が2つあった。

1. **候補ごとに1回 forward**（6〜8候補なら6〜8回）→ 全候補を右パディングして**1回のバッチ forward**にまとめた。右パディング＋causal attention なら実トークンのスコアは不変（**素朴な単一配列の計算と一致**を検証済み）。約2.5倍。
2. **全候補で同じ文脈を毎回再計算**していた → 文脈を**1回だけ forward して KV キャッシュ化**し、それを候補数ぶんタイル複製、各候補は「残りトークンぶん」だけ前進。長文脈で **~0.25s → ~0.16s**。

検証時に、新版が**素朴な真値（候補ごと単一 forward）と完全一致**することを確認（むしろ旧バッチ版より浮動小数的に忠実だった）。これで上限つき同期待ちが現実的になった。

## 11. 自分の語彙に寄せる — 適応学習

静的モデルの限界（16〜19/30）とは別に、**日常使用で一番効くのは適応**。Google日本語入力等が賢いのも結局これ。

- 確定するたび、IME が `reading→surface` をサービスに送って記録（`learned.json`）。
- 同じ読みで**2回以上**選ばれた表記を、辞書・LLM の上に**先頭へ昇格**。1回だけの選択は無視（文脈変換を壊さない）、長文（>12字）は学習しない（再利用されず肥大するだけ）。
- 優先順位は **学習 ＞ ユーザー辞書 ＞ LLM ＞ 辞書**。

トレードオフ: 読みのみの学習なので、一度2回選ぶと**文脈に関わらずその表記が #1** になる（雨/飴のような文脈依存語では稀に1回 Space が要る）。これは後で文脈つき学習で解決する（→ 18章）。

## 12. ユーザー辞書とホットリロード

辞書に無い語・略語を自分で登録できるようにした。

- `userdict.json`（`{"よみ":["表記",...]}`）を候補の**2番目に差し込む**（#1 は据え置き＝文脈変換を壊さない）。
- **保存すれば即反映**（ファイルの mtime を見てホットリロード、再起動不要）。
- 初期登録: AI/LLM/API/UI 等の略語、KYC・メルコイン等の社内語、そして序盤（5章）で「候補に出ない」と分かっていた口語略語（`おねしゃす→お願いします`、`とりま→とりあえず`）。**候補内27/30の残り3つ**をここで回収。

## 13. Sumibi風UX — ローマ字preedit と空白区切り

Sumibi（Emacs上のクラウドLLM日本語入力）の「ローマ字を打って一括変換」の操作感を、IME 側で近づけた。

- **入力中の表示をかな→生ローマ字に**（打った通り `kyou` と下線表示。Backspace も1文字単位）。
- **Space = 空白挿入／Shift+Space = 変換**。空白で単語を区切れる。
- **空白入りは区切り変換**: 空白で分割し、各セグメントを独立変換して連結。これは **dict-only**（手動で区切ると各セグメントが短くなり、辞書の頻度 top1 が速くて良い。LLM は孤立語だと外し<sup>※</sup>、かつ40倍遅い ※ `せいせい`→`聖性` など）。
- **略語**: 大文字始まりの英字はそのまま確定（`AI`→AI、`LLM`→LLM）。子音だけの略語は「かなにならない」ので自動でラテンが残る。

IME はアプリの確定済みテキストを後から触れないので、Sumibi の「下線なしの実テキストを領域変換」は原理的に無理。preedit で近似している（→ 17章）。

## 14. メニューと管理API

入力ソースアイコンのメニュー（`menu()` override）に管理機能を出した。

- **学習内容を確認**（`GET /learned` を整形して開く）
- **学習をリセット**（`POST /reset_learn`、全消し or `{reading}` で個別）
- **ユーザー辞書を編集…**（`userdict.json` を開く、保存でホットリロード）

## 15. モデル選定 — Qwen3-1.7B

高速化で品質寄りに振れるようになったので測り直した（新パイプラインの per-convert リランク時間 / 30問正解）。

| モデル | 速度（短〜長文脈） | 品質 | 常駐 |
|---|---|---|---|
| Qwen2.5-7B-Instruct | 220〜685ms | **19/30** | 4.3GB |
| **Qwen3-1.7B（採用）** | 96〜279ms | 17/30 | 1.2GB |
| Qwen2.5-1.5B-base（旧） | 57〜152ms | 16/30 | 1.0GB |
| Qwen2.5-0.5B | 26〜56ms | 14/30（辞書未満） | 小 |

7B は最高品質だが長文で 0.5〜0.7秒かかり**上限400msを超える**（＝長文で辞書にフォールバックして旨味が消える）。**Qwen3-1.7B が deadline 内に収まりつつ品質を1段上げる本命**として採用（warm ~0.13〜0.30s）。最高品質を取りに行くなら 7B＋deadline ~750ms に。

## 16. Sumibi との違い（実装比較）

同じ「LLMで日本語入力」でも設計はほぼ正反対。

| | Sumibi | RomKana |
|---|---|---|
| 変換 | **クラウドLLMが生成**（GPT/Gemini等） | **ローカル辞書が生成＋ローカル小型LLMが再ランク**（生成しない） |
| 動作環境 | **Emacs専用**（Elisp） | **macOS全アプリ**（Swift IMK＋Python） |
| 入力 | モードレス、実テキストを領域変換 | IME preedit（下線つき） |
| 学習 | なし | あり（適応学習＋ユーザー辞書） |
| オフライン/コスト | 要ネット・課金 | 完全ローカル・無料・私的 |
| 品質/速度 | 高品質・遅い（~1〜3s） | そこそこ・速い（~0.1〜0.3s） |

**速度差の本質は cloud/local ではなく generate vs score**。生成は自己回帰（出力トークンを1個ずつ＝数十回の逐次 forward）だが、RomKana の再ランクは**1パスのスコアリング**。RomKana が「生成は辞書に任せ、LLM は確率計算だけ」にしたのは、品質だけでなく速度の理由でもあった。

## 17. 「空白で区切れば正解率は上がるか」— 検証

直感的には上がりそうだが、**測ると単純には上がらなかった**（誤りの種類を交換するだけ）。

```
くるまでまつ
  whole(LLM)        : くるまで待つ      ← 「待つ」は正しい
  seg くるま/で/まつ : 車で末            ← 「車で」は直るが「まつ→末」に悪化
きしゃのきしゃはきしゃできしゃした
  seg きしゃの/きしゃは/…  : 記者の記者は記者で帰社した ← 同音語は区切っても割り直せない
```

理由: ①区切り変換は dict-only で**LLMの文脈補正が消える**、②孤立した短いセグメントは**辞書の頻度top1が支配**して `まつ→待つ` ではなく `末` のような誤りが出る、③同音語は語内部の曖昧で**切っても解けない**（文脈が要る）。

**結論**: 区切りは「全体変換が境界を誤った時の手動オーバーライド」としては有効だが、**正解率を底上げするレバーではない**。底上げを狙うなら「区切り」より「モデル強化」か「文脈つき学習」。

## 18. 適応学習を文脈つきに — 計測が設計を変えた

11章の学習にはトレードオフが残っていた。読みのみで覚えるので、一度覚えると**文脈に関わらずその表記が #1** になり、雨/飴のような文脈依存語の出し分けを潰す。これを直す。

**最初の発想（スコア加点）**: 学習を「#1へのハード昇格」ではなく、LLM の対数尤度に**加点して合算**する。そうすれば雨文脈では LLM が強く雨を推して学習加点を上回り、中立文脈では学習が勝つ——はずだった。

**計測したら前提が崩れた**: 1.7B の実スコアを見ると、この手の同音語の文脈分岐が**そもそも弱い**。

```
あめ  ctx「こどもがよろこぶ」(飴のはず): 天0.65 雨0.33 飴~0   ← 飴は最下位
きしゃ ctx「えきでんしゃと」(電車のはず): 記者0.38 汽車0.07     ← 汽車も最下位
```

LLM が「飴」「汽車」を**正しい文脈でも確信を持って却下**している。つまり「LLM信号を尊重して加点」方式だと、**ユーザーがいくら選んでも学習が昇格をブロックされる**＝逆効果。スコア加点は却下。

**採用（文脈バケツ＋広域フォールバック）**: LLM に頼らず、**ユーザー自身の「文脈→選択」履歴**で分岐する。

- 学習キーを `"<直前2文字>\t<読み>"` に。空バケツは「広域（どの文脈でも）」＝従来の振る舞い。
- 確定時に文脈も送り、広域と文脈別の**両方をカウント**。
- 変換は **文脈一致を優先 → 無ければ広域にフォールバック**。

これで同じ読みが文脈で出し分く（実測: 雨文脈→雨／飴文脈→飴／新規文脈→広域）。**LLM の弱さと無関係**で、疎性は広域フォールバックが吸収するので従来挙動も壊さない。旧 `learned.json`（flat形式）は起動時に自動移行。

**教訓**: 「LLM が賢いはず」という前提で設計しかけたが、**実スコアを測ったら逆**だった。賢いのは“ユーザーの癖”の方で、モデルではなかった。計測が設計を救った。

## 19. 全体を通しての教訓

1. **IME 開発の難所は変換ではなく OS への登録**（`.inputmethod.` 規約、署名は無関係）
2. **辞書(Mozc)は強い**。文脈無しの素直な文なら十分で、候補内率27/30
3. **オフザシェルフ LLM は、指示で選ばせる/生成させる限り、サイズに関係なく辞書を超えない**
4. **本当の正解は「LLMに賢く振る舞わせる」のではなく「LLMの確率計算という本業だけを使う」**こと（スコアリング方式）。sum で初めて辞書超え
5. **モデルはタスクとの相性が全て**: SELECT が得意な Sarashina は scoring が苦手、scoring は Qwen 系が得意
6. **早すぎる棄却に注意**: 「LLMリランクは効かない」は弱いモデル＋mean での誤った結論だった
7. **UXのトレードは直視する**: 「待ち時間ゼロ」と「1番目が動かない」は両立しない。逃げず「待ちを十分小さくして同期」で解いた
8. **速度の本質は generate vs score**: 遅さの主因は cloud/local よりも生成の有無。スコアリングは1パスで速い
9. **無駄な再計算を疑う**: バッチ化＋文脈KVキャッシュで約2.5倍。「同じものを何度計算しているか」を見る
10. **静的モデルの限界の外側に学習がある**: 日常精度を一番上げるのは適応学習＋ユーザー辞書
11. **実測がすべて**: 「このモデルが良いはず」「区切れば正解率が上がるはず」はどちらも外れた。ベンチで確かめる
12. **賢いのはモデルとは限らない**: 同音語の文脈分岐は、弱い1.7Bより**ユーザー自身の履歴**の方が当たる。学習をLLMに従属させず、独立に効かせる設計が正解だった
