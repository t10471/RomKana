# 自作 macOS IME「RomKana」開発記

ローマ字を打つと、文脈を見てかな漢字交じり文に変換してくれる macOS の入力メソッド（IME）を、**完全ローカル**で自作しました。「変換の賢さは、手元で動くLLMでどこまで上げられるんだろう？」というのが知りたくて、いろいろ測りながら作っていった記録です。最終的には、かな漢字変換に特化したニューラルモデルへ乗り換えました。

例: `AI de nihongo wo henkan suru`（空白で区切って打つ）→ `AIで日本語を変換する`

**きっかけ**は [X のあるポスト](https://x.com/fta7/status/2059754329058488795)でした。気になって調べてみたら、ローマ字をLLMでかな漢字に変換する **[Sumibi（炭火）](https://github.com/kiyoka/Sumibi)** という先人がいました。ただ Emacs 向けだったので、「macOS のどのアプリでも同じことがしたいな」と思い、IME として自分で作ってみることにしました。

話は大きく2部に分かれます。

- **第1部（1〜7章）**: 辞書（mozcpy）＋ローカルLLMのスコアリングで、「LLMで変換を賢くできるか」をひたすら測っていた時期です。
- **第2部（8章〜）**: 変換エンジンを **AzooKeyKanaKanjiConverter ＋ Zenzai（zenz ニューラルモデル）** に作り直し、Python をやめて単一プロセスにした話です。

現行構成のリファレンスは [`architecture.md`](https://github.com/t10471/RomKana/blob/master/docs/architecture.md)、旧構成は [`architecture-legacy.md`](https://github.com/t10471/RomKana/blob/master/docs/architecture-legacy.md) にまとめています。

---

## 1. 何を作ったか

- **RomKana.app**: Swift + InputMethodKit で書いた、macOS 用の本物の IME です。システム設定の入力ソースにちゃんと出てきます（配布先は `~/Library/Input Methods/`）。
- **変換**: 最初は辞書（mozcpy）＋お好みでローカルLLMの再ランク（Python の常駐サービス）。今は AzooKey + Zenzai を **IME のプロセスの中**で直接動かしています。
- 打っている間は、ローマ字をローカルですぐに表示します。重い変換が走るのは Space を押したときだけです。このおかげで、入力中に固まりません。

---

## 2. 最初の山 — IME が OS に出てこない

いきなりですが、最初の最大の難所は変換ではありませんでした。**「作った IME が入力ソースの一覧に出てこない」**のです。署名を変えたり、何度も再ログインしたり、ログを追ったり……それでも出てこなくて、ここで数日溶かしました。

**真因はバンドルIDでした。** macOS は `CFBundleIdentifier` に **`.inputmethod.` という文字列が入っていること**を要求します。

- ❌ `com.toshinao.romkana` → 出ない
- ✅ `com.toshinao.inputmethod.RomKana` → 出る（Apple 純正も `com.apple.inputmethod.Kotoeri` ですね）

ちなみに犯人だと疑ったのに**全部シロ**だったものも書いておきます。署名・公証は無関係（ad-hoc 署名で普通に動きます）、キャッシュ汚染でもなく、`ComponentInputModeDict` も必須ではありませんでした。**IME 開発でいちばん難しいのは「変換」ではなく「OS にどう認識させるか」**だった、というのが最初の学びです。

---

## 3. 変換エンジンの土台は辞書（mozcpy）

最初はローカルLLM単体（LFM2.5-1.2B）に変換させてみたのですが、文脈を無視したり、チャットの返事が混ざったりで、とても実用になりませんでした。そこで **mozcpy（Mozc 派生の辞書変換）を主役**に据えました。速い（~20ms）し、安定しているし、実在する語しか出しません。

ここは実際に打ってみて出たバグを地道に潰す作業でした。たとえば、末尾の単独 `n` を「ん」に確定していなくて `shinbun` が崩壊したり、外来音の `じぇ ちぇ てぃ` が変換表になくて `プロジェクト` が `ぷろzyぇくと` になったり。打つたびに直していきました。

---

## 4. 本題 — ローカルLLMで変換を賢くできるか

ここからが本題です。同音異義語を文脈で正しく変換できるか、いろんなモデルでひたすら測りました。

**テストセット**は、「**同じ読みでも、文脈しだいで漢字が変わる**」同音異義語を中心に集めました。易しめが10問、本格的なものが30問です。

たとえば「あめ」という読みは、前後の言葉で正解が変わります。

- 「そらがくもってあめがふる」なら → **雨**
- 「こどもがあめをなめる」なら → **飴**

同じように「きしゃ」→ 記者／汽車、「せいさん」→ 精算／清算、「いし」→ 意思／医師 のように、読みは一緒でも文脈で正解が分かれる組を並べました。1問は「かなの読み（多くは短い文）」で、変換器がこの出し分けをどれだけ当てられるかを見ます。

測っていて気づいたのが、**指標は top-1 ではない**ということでした。IME はポップアップから選べるので、本当に大事なのは「**正解が候補一覧に入っているか**」です。

- 辞書の top-1 は **15/30**。
- でも**正解が候補のどこかに入っている率は 27/30**（残り3つは、読み自体が辞書にない略語でした）。
- つまり辞書＋ポップアップで、実用上はかなり良かったわけです。top-1 の数字だけ見ると、実態より厳しく見えていました。

そのうえで、いろんなモデルを横断で測ってみました（難所30問・SELECT＝候補から選ばせる方式）。

| モデル | サイズ | SELECT正解 | 速度 |
|---|---|---|---|
| mozcpy（辞書） | 小 | 15/30（候補内27） | ~20ms |
| Qwen2.5-7B | 4.3GB | 15-16/30 | ~1000ms |
| Qwen3-1.7B | 1GB | 15/30 | ~480ms |
| Sarashina2.2-3B | 2GB | 13-14/30 | ~600ms |
| LFM2.5-1.2B | 0.8GB | 1/30 | ~200ms |

結果は、ちょっと拍子抜けでした。**指示で選ばせる・生成させるやり方だと、大きいモデルでも小さいモデルでも、辞書（15）を超えません。** 賢いはずのモデルでも「辞書の上位候補をなぞるだけ」なのです。ついでにクラウドの Opus 4.8 も測ったら ~29/30 で、文脈推論の地力が桁違いでした。

## 5. ファインチューニングも試してみた（Sarashina-3B + LoRA）

「汎用がダメなら専用化だ」と思って、MLX の LoRA で IME タスクに微調整してみました（約2,500件、テストの30問は使っていません）。

| 方式 | 素 → FT |
|---|---|
| GEN（生成） | 5/30 → **9/30**（+80%、効いた！） |
| SELECT（選択） | 13/30 → **9/30**（悪化…） |

**FT 自体は効きました**（生成が1.8倍）。ただ辞書（15）には届かず、生成型は「製産」「整算」みたいな造語をでっち上げる癖が残ります。そして面白い（というか痛い）のが、**SELECT を FT したら逆に悪化した**こと。原因はコーパス由来ラベルのノイズで、SELECT-FT はラベルの品質にものすごく敏感でした。「少しのデータで辞書超え」という見立ては、正直、楽観的すぎました。

---

## 6. ブレイクスルー — スコアリング方式

ここで発想を変えました。LLM に**指示で選ばせる・生成させる**のをやめて、LLM が本来得意な「**次に来る自然な日本語の確率を計算する**」ことだけをやらせます。

1. 候補を出すのは**辞書（mozcpy）に任せる**（実在する語だけ＝でっち上げが物理的にゼロ）。
2. それぞれの候補について `P(候補トークン | 文脈)` を **1回の forward** で計算する。
3. **合計対数尤度（sum）がいちばん高い**候補を選ぶ。

| モデル | SCORE-sum | 速度 |
|---|---|---|
| 辞書（基準） | 15/30 | ~20ms |
| Qwen2.5-7B | **19/30** | ~860ms |
| Qwen3-1.7B | **17/30** | ~410ms |
| Qwen2.5-1.5B-base | 16/30 | ~235ms |

**ここで初めて辞書（15）を超えました。** 生成しないので速いし、造語も出ません。ここが面白かったところを2つ。

- 🔑 **sum が正解で、mean はダメ**でした（mean だと11止まり）。自然で高頻度な語ほど少ないトークンにまとまるので、合計尤度がそれを味方につけてくれます。平均にするとその信号が消えてしまう。実は最初に「LLMリランクは効かない」と切り捨てたのは mean で測っていたからで、**早すぎる棄却**だったわけです。
- scoring に向くのは「ベースの確率が素直で、日本語の地力があるモデル」でした。だから Qwen が好相性。逆に Sarashina は SELECT だと13点なのに scoring だと5点と、相性がひっくり返りました。

## 7. 毎日使えるUXへ — トレードオフと向き合う

変換の芯ができたあとは、「毎日使える道具」に仕上げる作業でした。計測で考えが変わった判断を3つだけ。

- **フリーズ vs ジャンプ**。同期で変換すると Space のたびに固まります。かといって非同期で先に出すと、**1番目が後から入れ替わって**気持ち悪い（ジャンプ）。先頭を辞書で固定すると、今度は#1 が LLM の品質に戻らない。結局たどり着いたのは**上限つきの同期待ち**で、辞書と LLM を並行で走らせ、最大400msだけ待つ、という形でした。ここで学んだのは、**「待ち時間ゼロ」と「1番目が動かない」は同時には成立しない**ということ。逃げずにどちらかを選ぶしかないんですね。
- その同期待ちを成立させるために**高速化**もしました。候補ごとに forward していたのを、バッチ化と文脈の KV キャッシュで約2.5倍に。「同じ計算を何度もやっていないか？」を疑うと効きます。
- **適応学習で計測に救われました。** 最初は「LLM の尤度に学習を加点する」設計にしかけたのですが、実際のスコアを見たら、1.7B は同音語の文脈分岐が**そもそも弱い**。正しい文脈でも「飴」「汽車」を自信満々で却下してくるのです。なので LLM に頼るのをやめ、**ユーザー自身の「文脈→選んだ語」の履歴**で分岐させました。賢いのはモデルではなく、使う人の癖のほうでした。

# 第2部 — AzooKey + Zenzai への移行

第1部で 17〜26/30 までは来たのですが、壁が2つ残っていました。**(a) 汎用LLMは、どう工夫しても変換専用には届きにくい。(b) Python/MLX を別プロセスで常駐させるのが重い（~1.2GB）し、HTTP 往復や LaunchAgent の管理も地味に面倒。** そんなとき、かな漢字変換に特化したニューラル変換がローカルで動く選択肢を見つけて、エンジンごと作り直すことにしました。

## 8. なぜ作り直したか — zenz / Zenzai との出会い

- クラウド生成（Sumibi 系）は賢いけれど、ネットが要ります。完全ローカルの方針は曲げたくありませんでした。
- そこで見つけたのが **[zenz](https://huggingface.co/Miwa-Keita/zenz-v3.2-small-gguf)**（ensan さんのかな漢字変換特化モデル、CC-BY-SA）と、それを使う **[AzooKeyKanaKanjiConverter](https://github.com/azooKey/AzooKeyKanaKanjiConverter)**（純 Swift, MIT）です。**Zenzai** は [llama.cpp](https://github.com/ggml-org/llama.cpp) ＋ zenz の GGUF をローカルで回し、辞書変換の上に乗せて候補を賢く並べてくれます。
- 採用したのは **zenz-v3.2-small**（GPT-2系 char、約95M、量子化後で **70MB**）。旧構成の Qwen3-1.7B（~1.2GB）とは桁が違う軽さです。

## 9. 単一プロセスにする

変換が純 Swift ＋軽い GGUF で完結するので、もう別プロセスにする理由がなくなりました。HTTP サービスも LaunchAgent も MLX も全部捨てて、変換を IME のプロセスの中に入れました。`KanaKanjiConverter` は**静的に1つだけ共有**して、zenz の GGUF と Metal のロードをプロセスで一度きりにしています。常駐メモリは **~1.2GB → ~74MB**。初回ロードだけ ~1〜2秒かかるので、`activateServer` でダミー変換して**ウォームアップ**しておけば、あとの変換は ~50ms で返ってきます。

## 10. 同じ30問で旧と新を測ってみた（正直な結果）

移行したあと「あれ、精度が上がった気がしないな…」と感じたので、旧実装を消さずに残しておいて、**同じ30問で正面から比較**しました。

| 方式 | 正解 | 速度 | メモリ |
|---|---|---|---|
| 旧: mozcpy 辞書のみ | 22/30 | ~20ms | — |
| 旧: mozcpy + Qwen3-1.7B | 26/30 | ~100–280ms | ~1.2GB |
| **現: AzooKey + Zenzai** | **28/30** | **~50ms** | **74MB** |

全体では新しいほうが上でした。ただ、**割れた問題のうち1つは旧が勝っていました。**「けっきょくせいどがよくなった」が、旧Qwen は**精度**○、現Zenzai は**制度**✗。よりによって自分が実際に打った文で外したわけで、広い文脈の理解は 1.7B のほうが強い場面が残るんだな、と正直に受け止めました。とはいえ速度・メモリ・依存の少なさは新しいほうが圧勝なので、戻さず、このまま弱点を潰していくことにしました。

## 11. 候補の出し方を作り直す

- **ゴミ候補が多い問題。** `mainResults` には「文まるごと」だけでなく「途中までの文節候補」も混ざってきます（精度が／精度／制度…）。そこで**入力全体をカバーする候補（`correspondingCount == 入力長`）を先頭**に出し、部分候補は後ろに回しました。
- **逆に候補が少なすぎる問題。** 絞りすぎると、今度は同音異義を選び直せません。ここで効いたのが、ユーザーが**空白で区切ってくれる＝文節の境界を教えてくれている**こと。チャンクごとに上位候補を取って、「**1チャンクだけ別の候補に差し替えた文**」を並べました。これで `制度が上がらない…` の次に `精度が上がらない…` が出ます。文はキレイなまま、1語だけ選び直せるようになりました。

## 12. 学習・ユーザー辞書・設定

- **学習**は AzooKey 組み込みのものを使いました。確定した `Candidate` を `updateLearningData` と `setCompletedData` に渡すだけです（`learningType: .inputAndOutput`、データは App Support に保存）。第1部で自作した文脈バケツ学習より、こちらのほうが精度が良かったので任せました。
- **ユーザー辞書**は `userdict.json`（`{"読み":["表記"]}`）を読み、カタカナのルビ（AzooKey の `ruby` フィールド＝読みのこと）に直して動的辞書へ流します。
- **設定**は、候補数・推論上限・学習ON/OFF・辞書の重みといったチューニング値を **`config.json`** に出しました。ファイルがなければ自動で作られるので、再ビルドなしでいじれます。

## 教訓

1. **IME 開発でいちばん難しいのは変換ではなく OS への登録**でした（`.inputmethod.` の規約）。
2. **辞書は強い。** 素直な文なら十分で、候補内包率は 27/30。
3. **汎用LLMは、指示で使う限りサイズに関係なく辞書を超えません。** 本領は「賢く振る舞わせる」ではなく「確率計算という本業だけ使う」こと（sum スコアリングで初めて辞書超え）。
4. **モデルはタスクとの相性が全て**でした（SELECT 得意な Sarashina が scoring だと苦手）。そして**早すぎる棄却に注意**（mean での誤判定）。
5. **UX のトレードは直視するしかない。**「待ち0」と「1番目が動かない」は両立しません。
6. **賢いのはモデルとは限らない。** 同音語の文脈分岐は、ユーザーの履歴のほうが当たりました。
7. **専用モデル ＞ 汎用LLMの工夫。** かな漢字変換は、特化モデル（zenz）をローカルで回すほうが素直に強く、軽く、速い。
8. **評価は感覚ではなく同一条件の実測で。** 旧を残して同じ30問で比べたから、「全体は勝ち・でも一部は負け」という正直な姿が見えました。

## 付録: 主要ファイル（現行）

- `Sources/RomKanaController.swift` — IMKInputController。入力・変換・候補・確定・学習・辞書・設定・メニュー。Space=空白／Shift+Space=変換、区切り変換（チャンク別の代替候補）、文まるごと変換（全体カバー優先）。
- `Sources/RomajiConverter.swift` — ローカルの romaji→かな変換表。
- `Sources/Config.swift` — `config.json` の読み込みと既定生成。
- `Package.swift` — `AzooKeyKanaKanjiConverter`（trait `Zenzai`）に依存、C++ 相互運用あり。
- `models/zenz-v3.2-small-gguf/ggml-model-Q5_K_M.gguf` — 同梱する zenz モデル。
- `scripts/build_install.sh` — `swift build` → バンドル組立（＋`llama.framework`・`Dictionary`・GGUF）→ 署名 → 再読込。
- データ — `~/Library/Application Support/RomKana/`: `config.json` / `userdict.json` / AzooKey の学習メモリ。
